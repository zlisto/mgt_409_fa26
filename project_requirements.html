Final Project: Executive Co-Pilot

Benchmark edition — integration + prompts + automated tests (no Demo Day)

One sentence. Build an Executive Co-Pilot that routes business tasks to the right agent modules, with prompts in a prompts/ folder, and pass the course benchmark suite.

Homeworks teach individual powers (extract, RAG, research, bots, MCP, deploy). The final project is the exam: one unified system that must survive instructor-run tests — especially a hidden benchmark set you have not seen before submit.

Homework vs final project

Homeworks (6) Final project
Goal Learn one power at a time Compose powers into one product
Scope Narrow, creative, industry-specific Fixed spec + benchmark tests
Grading Features + short memo Pass rate on tests + integration + prompts

System architecture

Your submission should behave like a single co-pilot, not six disconnected homework zips:

User question / file upload
        ↓
   Router agent (prompt decides: extract | research | KB | decide)
        ↓
   Module(s) + tools + RAG
        ↓
   Structured response + citations + logs

You may extend your Homework 6 Vite app and deployed URL. You do not need to start from scratch.

Required repository structure

This layout makes benchmarking and grading reproducible:

final_project/
├── README.md              # how to run, env vars, live URL
├── prompts/
│   ├── router.txt         # routes tasks to modules
│   ├── pdf_extract.txt
│   ├── image_extract.txt
│   ├── research.txt
│   ├── rag_qa.txt
│   └── decision.txt       # multi-step / structured memo
├── app/                   # Vite frontend and/or bot entrypoint
├── agents/                # module code
├── requirements.txt
└── self_report.json       # optional: your public-suite run before submit

Prompts are graded by outcomes. If pdf_extract.txt fails PDF tests, the prompt and wiring failed — not just a stray bug.

What the benchmarks test

Benchmarks map directly to course skills. You will receive a public practice suite before submit; the instructor runs an additional hidden suite on every submission.

Bucket Course skill Example test
B1 — PDF → JSON Lectures 1–2 Given a 10-K snippet, extract {revenue, fiscal_year, top_risk} within schema
B2 — Image → JSON Lecture 3 Given a receipt or ad image, extract required fields
B3 — RAG + cite / refuse Lectures 5–6 KB questions must cite a source; trap questions must refuse or escalate
B4 — Web research Lecture 4 “Who are 3 competitors in X?” → structured table with ≥3 source URLs
B5 — Agentic decision Lectures 7–8 Scenario → valid JSON matrix + executive summary; schema must validate
B6 — Integration Lectures 9–12 One entrypoint handles ≥3 task types without manual module picking
B7 — Ops (optional bonus) Lectures 9–10 Telegram/Slack response or MCP tool call appears in logged trace

Public practice benchmarks (sample)

Run these locally before you submit. Hidden tests follow the same format with different inputs.

  • pdf_01 — Extract revenue and fiscal year from a short 10-K excerpt; JSON schema validation
  • image_01 — Extract vendor, total, and date from a receipt image
  • rag_01 — Answer from handbook; must cite employee_handbook.pdf
  • rag_trap_01 — Question not in KB; must refuse or hand off (no fabricated policy)
  • research_01 — Three competitors in a named niche; table + ≥3 URLs
  • decision_01 — Vendor selection scenario; JSON comparison matrix + 150-word exec summary
  • router_01 — Upload PDF vs ask KB question vs research query; correct module invoked

Practice command (published before Lecture 13):

python run_benchmarks.py --repo . --suite public

Each test is a JSON spec. Example:

{
  "id": "rag_trap_01",
  "type": "rag_qa",
  "input": {"question": "What is our unlimited refund policy?"},
  "assert": {
    "must_refuse": true,
    "must_not_contain": ["unlimited refund", "100% guarantee"]
  }
}

Deliverables

  1. GitHub repository — required; clear README; no API keys in repo
  2. prompts/ folder — required; one file per module plus router
  3. Live URL — strongly encouraged (Vercel or equivalent from Homework 6)
  4. Executive memo — 1 page max: who uses the co-pilot, one workflow, estimated cost per 100 runs (tokenomics)
  5. No live presentation — Lecture 14 is instructor-led; there is no Demo Day

Grading (100 points)

Component Pts How it is graded
Hidden benchmarks 50 Automated pass/fail on instructor hidden suite
Public benchmarks 10 Sanity check; confirms you did not overfit to leaked cases
Prompt quality (outcome-based) 15 Partial credit when modules run but cite, refuse, or structure incorrectly
Integration & architecture 15 Router works; shared config; readable logs
Deploy & reproducibility 5 README runs; env documented; live link works if provided
Executive memo 5 Business sense and cost awareness

Optional bonus (not required for full credit)

  • Telegram or Slack bot passes the same benchmarks via chat
  • MCP connector used in at least one workflow (logged trace)
  • Token budget mode: answer under a stated cost ceiling per query
