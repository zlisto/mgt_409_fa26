Homework 1: Small Business Valuation Agent

Scenario

Spoke & Wrench Bicycle Repair is a small New Haven shop. The owner keeps messy records—PDF receipts, bank and card statements, a lease, emails, and handwritten notes—not a clean QuickBooks export. Download the document pack (hw1_spoke_and_wrench.zip).

Your job is to build an agent that reads these documents at runtime. The documents disagree, overlap, and leave things out. Your Python code should extract text from each PDF, send that text (with your prompts) to the OpenAI API, get structured JSON back, then chain those JSON files through later agent steps for reconciliation, judgment calls, valuation, and memos. Grades reward defensible accounting judgment and a working LLM pipeline—not a script that prints pre-written answers.

Agent architecture (required)

agent.py orchestrates a multi-step pipeline. You may organize the code in one file or several helper modules, but each analytical step below must call the OpenAI API using the matching prompt file from prompts/. Local PDF text extraction (pypdf, pdfplumber, etc.) is fine—that is preprocessing before the LLM sees the document. 

Document pack (PDFs + emails)
        │
        ▼
  [1] Extract ── prompts/pdf_extract.txt, prompts/email_extract.txt
        │         One API call per PDF (or small batch) → receipts, bank, card JSON
        ▼
  [2] Reconcile ── prompts/reconcile.txt
        │           API call with Problems 2–3 JSON + relevant emails → reconciliation_log.json
        ▼
  [3] Judgment ── prompts/judgment_calls.txt
        │          API call on hardest rows → judgment_calls.json
        ▼
  [4] P&L ── roll up reconciliation_log.json → income_statement_jan2026.json
        │         (Python math is fine; data must come from step 2)
        ▼
  [5] Value ── prompts/valuation.txt
        │        API call with income statement + emails + IOU/lease text → valuation_inputs.json
        ▼
  [6] Memos ── prompts/memo_buyer.txt, prompts/memo_bank.txt
              API calls → memo_buyer.md, memo_bank.md

report.py (no API) reads output/ JSON → valuation.html

Not allowed: hardcoding Spoke & Wrench dollar amounts, vendor names, or reconciliation resolutions as Python literals and skipping the API. Using the OpenAI API only to help you write the code, while the submitted script never calls it at runtime, does not satisfy the assignment.

Download data: hw1_spoke_and_wrench.zip (unzip before running your agent).

API key: set OPENAI_API_KEY in your environment (see course instructions). Do not put keys in your zip.

Submit: one zip file, hw1.zip — see Problem 10.

Problem 1 — Set up your agent project (10 points)

Turn in a Python project we can run from the terminal:

  • agent.py — runs the full LLM pipeline above and writes JSON + memos to output/
  • report.py — builds output/valuation.html from JSON (no API required here)
  • requirements.txt — include openai and your PDF library
  • README.md — install steps, how to set OPENAI_API_KEY, and the exact commands below
  • prompts/ — plain-text system/user instructions your code loads and sends to the OpenAI API at each step (see Problem 10 file tree)

We will run (with our own API key in the environment):

python agent.py --docs-dir PATH --out-dir output then python report.py --json-dir output --out output/valuation.html.

Both commands must complete without errors and produce fresh output from the document pack—not from values frozen in your source code. Your README should name the OpenAI model you use (e.g. gpt-4.1-mini) and briefly describe how each prompt file is used.

Problem 2 — Receipt ledger (12 points)

Agent step: for each purchase receipt PDF, extract text locally, then call the OpenAI API with prompts/pdf_extract.txt (or a receipt-specific variant you document) and save the combined result as output/receipts.json.

Documents: Park Tool, New Haven Bike Parts, Home Depot, Speedy Courier, Amazon headphones, and the eBay repair-stand confirmation. These are proof-of-purchase PDFs—not the bank statement, credit card statement, lease, IOU, or Hartford Mutual quote.

For each line item, record vendor, date, description, amount_usd, category (e.g. tools_equipment, shipping, personal_electronics), and source_file. If a field is missing from the PDF, list it under fields_not_found—tell the model not to invent values in your prompt.

The receipt ledger is a catalog of what each document says. Do not decide here whether something is business vs personal or whether it hits January profit and loss—that is Problems 3 and 4.

Problem 3 — Bank and credit card classification (12 points)

Agent step: extract text from bank_statement_jan2026.pdf and credit_card_jan2026.pdf, call the OpenAI API with prompts/pdf_extract.txt (tuned for statements), and save output/bank_transactions.json and output/credit_card_transactions.json.

Turn each posted line into structured data for later reconciliation and the January profit and loss statement (P&L)—also called an income statement: revenue minus expenses for the month.

Bank transactions. For each deposit or withdrawal, record the date, description, amount, whether it is business or personal, and how you would book it (revenue, expense, owner draw, and so on).

Credit card transactions. For each charge, record the date, merchant, amount, whether it is business or personal, and—if business—what type of shop expense it is.

Both files — every row:

  • classification: business or personal

Bank rows — also include:

  • direction: credit (deposit) or debit (withdrawal)
  • accounting_label — for credits: revenue, owner_contribution, transfer, or other; for debits: expense, owner_draw, transfer, or other
  • expense_type — when accounting_label is expense, a short label (e.g. rent, utilities, cogs_parts)

Credit card rows — also include:

  • expense_category — for rows you classified business (e.g. shipping, tools_equipment); omit or set null for personal rows

Your prompt should tell the model that issuer categories on the card statement (e.g. “Grocery stores,” “Gas stations”) are automated MCC labels—they do not by themselves mean business vs personal. In step 2 you classified from the statement alone; step 4 will reconcile against receipts and email_owner_voice_memo.txt.

Problem 4 — Reconciliation log (18 points)

Agent step: call the OpenAI API with prompts/reconcile.txt. Pass in your receipts.json, bank_transactions.json, credit_card_transactions.json, and the text of relevant emails and other PDFs (lease, IOU, Hartford quote, owner memo). Save the model’s structured output as output/reconciliation_log.json.

This is the heart of the assignment. List amounts that appear in more than one place, or that need a single P&L decision after cross-checking sources. Some rows will have unclear categories or different dollar amounts across documents—record what each source shows, whether it belongs in January P&L, and the single amount you will use. For each item include:

  • id — short label you choose
  • sources — file names where you saw it
  • amounts_seen — what each source shows (keys should be file names)
  • included_in_january_p_and_l — yes if this item is a business revenue or expense for January 2026 that belongs on your income statement (Problem 6); no if you exclude it (e.g. personal spending, future pipeline work, or off-books items not in January books)
  • amount_used_in_p_and_l — the single dollar amount you count in January P&L when included_in_january_p_and_l is yes; use 0 when no
  • resolution — plain English: what you did and why

Your reconcile prompt should require valid JSON only and tell the model to cite source files, not invent amounts.

Problem 5 — Judgment calls (10 points)

Agent step: call the OpenAI API with prompts/judgment_calls.txt, passing reconciliation_log.json and supporting document excerpts. Save output/judgment_calls.json.

Some reconciliation rows need a judgment call—the documents do not settle on their own whether an amount belongs in January P&L or how much to count. Pick three rows where the agent (or you, reviewing its draft) found it hardest to set included_in_january_p_and_l or amount_used_in_p_and_l, and explain why.

For each of the three judgment calls, include:

  • transaction_id — use the id from that row in your reconciliation log
  • included_in_january_p_and_l — your decision: yes or no
  • amount_used_in_p_and_l — the dollar amount you count in January P&L, or 0 if excluded
  • evidence_for — bullet list of facts and document names that support your decision
  • confidence — how sure you are: high, medium, or low (at least one of your three calls must be low)

You may edit the model’s three picks before saving if you disagree—but document your final reasoning in evidence_for.

Problem 6 — January income statement (10 points)

Build the shop’s January 2026 profit and loss (P&L)—also called an income statement: total business revenue, expense lines, and net income. Save the result as output/income_statement_jan2026.json.

Build your income statement from reconciliation_log.json (Problem 4). For each row where included_in_january_p_and_l is yes, use amount_used_in_p_and_l and roll it into revenue or expense lines. This roll-up can be plain Python—no API call required—but the reconciliation input must come from your LLM reconcile step, not hardcoded rows.

Your JSON must include:

  • period — 2026-01
  • revenue_usd — total January business revenue
  • expense_lines — array of expense rows; each row needs:
    • label — short name for the line (e.g. rent, card_tools_equipment)
    • amount_usd — dollars for this line
    • category — expense type (e.g. rent_occupancy, tools_equipment, cogs_parts, shipping)
    • sources — file names backing this line (e.g. bank_statement_jan2026.pdf, credit_card_jan2026.pdf)
  • total_expenses_usd — sum of all expense lines
  • net_income_usd — revenue_usd minus total_expenses_usd

Problem 7 — Valuation scenarios (12 points)

Agent step: call the OpenAI API with prompts/valuation.txt. Pass income_statement_jan2026.json, pipeline/customer emails, IOU and lease text, and your reconciliation notes. Save output/valuation_inputs.json.

You have rebuilt January books. The next question is different: what is this business worth? There is no single correct number. Value depends on who is asking and what they believe about future cash the shop can generate.

For a small shop like this, a common approach is annualized profit × a multiplier. Take January net income from Problem 6, multiply by 12 for a rough full-year earnings figure, then apply a multiple (e.g. 2×–3.5×) to get a purchase price range. In each scenario, low_usd and high_usd are what someone might pay to buy the whole business—not monthly or annual profit. Explain your multiple and any add-ons in method.

Different readers trust the numbers differently. A buyer weighs earnings quality, future jobs not yet collected, and downside risk. A bank focuses on reliable cash flow, collateral, and personal spending mixed with the business. Your three scenarios— books_only, adjusted, and skeptical_buyer—should reflect those different levels of trust.

Required sections in valuation_inputs.json:

  • booked_january_net_income_usd — net_income_usd from your January income statement (Problem 6)
  • annualized_net_income_usd — that figure × 12; the earnings base for your multiplier math
  • off_books_receivables — money customers already owe for work completed but not yet deposited (e.g. an IOU or unpaid invoice)
  • pipeline_orders — future jobs from customer emails (not January revenue); note when sources disagree on price or timing
  • excluded_from_valuation — intangibles, missing documents, or risks a buyer or lender would still ask about
  • valuation_scenarios — three scenario objects. Each includes:
    • low_usd — the low end of what someone might pay to buy the shop
    • high_usd — the high end of that purchase-price range
    • method — plain English showing your math: annualized net income, multiple, and any receivables, pipeline, or discounts you added
    Name the three scenarios:
    • books_only — annualized net income × multiple only
    • adjusted — add receivables and probability-weighted pipeline
    • skeptical_buyer — lower multiple, heavier discounts
  • assumptions — pipeline probabilities and other shared judgments (list once here); each scenario’s method shows how you applied them to get that scenario’s range

For adjusted, assign each pipeline order a probability between 0 and 1: your estimate that the job will happen and pay roughly the emailed amount (e.g. 0.75 probability on a $4,000 job would be counted as $3,000). Show the math in each scenario’s method. A single unexplained range is not enough.

Problem 8 — Valuation webpage (5 points)

Your JSON files hold the analysis; a buyer or banker will not read raw JSON. Turn your work into a one-page valuation briefing they can open in any browser—a readable summary of what you found and what you think the shop is worth.

report.py should read your output/ JSON files and build that page— no OpenAI API call required for this step. Save the result as output/valuation.html. Use the same numbers as your JSON; do not hand-edit totals in the HTML.

The page should include:

  • January P&L — revenue, expenses, and net income from your income statement
  • Personal vs business — bank and card rows you classified as personal (what you excluded from the shop books)
  • Pipeline orders — future jobs from Problem 7
  • Valuation scenarios — your three purchase-price ranges (books_only, adjusted, skeptical_buyer) with methods
  • Missing documents and risks — items from excluded_from_valuation

Problem 9 — Memos (8 points)

Agent step: call the OpenAI API with prompts/memo_buyer.txt and prompts/memo_bank.txt, passing your final JSON summaries. Save output/memo_buyer.md and output/memo_bank.md.

Your JSON and webpage summarize the numbers; a real deal still needs a written recommendation. Each memo should be about 250–400 words. You may lightly edit model output before saving, but the drafts must come from API calls—not copy-paste from chat outside your pipeline.

Buyer memo. You are advising a potential acquirer: How strong is January earnings? What pipeline orders add upside? What is the top risk if they buy the shop?

Bank memo. You are advising a lender considering a loan to the business: Is personal spending mixed with the shop? How large are owner draws? What collateral or financial detail is not in these PDFs?
